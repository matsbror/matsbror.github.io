The Map Pattern in Other Languages

In the first post on the map pattern I showed how the pattern could be coded in C++ in a few different ways. Now it’s time to look at some other popular languages: Rust, Java and Python.

A caveat: I am not a day-to-day user of these languages and they are partly a mystery to me as I do not exactly know how they are translated to machine code at the end. Rust should be pretty clear, but I need some more time to familiarise myself with it. Java and Python, however, are pretty obscure to me in terms of how the code gets translated into machine code. I have tried my best so please be nice in your comments and suggest improvements.

Also, the performance numbers here should be used only to compare versions within the same language. I may not have used the same arguments or the same computer between languages.

Rust

The map pattern is just as trivial in Rust as to use the Standard template library in C++ (see (previous blog post)[../1-map-pattern/]). Here is the code:

fn map_seq(arr: Vec<i64>, foo: fn(&i64) -> i64) -> Vec<i64> {
    arr.iter().map(foo).collect()
}

The map interface I use here takes a vector of numbers: arr, applies the map function fooon every element and returns the resulting vector. This becomes a one-liner in Rust since the language has a built in functionality for mapping a function on each element of a vector.

The Rayon crate makes it equally easy to make the above code parallel:

fn map_seq(arr: Vec<i64>, foo: fn(&i64) -> i64) -> Vec<i64> {
    arr.par_iter().map(foo).collect()
}

Note the par_iter()! It creates iterators working in parallel on threads in a thread pool. It makes it very easy to work on iterable objects in parallel. At the end, the iterables are joined back in a vector which is returned to the caller.

The sequenial version took: 0.9 s.

The parallel version took 0.4 s, yileding a speedup of ~2.2.

Java

Although I have written a fair amount of Java, I have not yet worked a lot in writing data-parallel Java code. My first attempt was using the Streams interface which has a parallel counter-part and the code looks very much like the Rust code:

  public List<Integer> map_seq_stream(List<Integer> arr, UnaryOperator<Integer> foo) {
    return arr.stream().map(foo).collect(Collectors.toList());
  }

  public List<Integer> map_par_stream(List<Integer> arr, UnaryOperator<Integer> foo) throws Exception {
      return pool.submit(() -> arr.stream().parallel().map(foo).collect(Collectors.toList())).get();
  }

These are very similar to the Rust versions above. The pool reference is a field I store in a class where map_par_stream is a method. It holds a reference to a thread pool I use to create a controlled number of threads for the parallel version.

Sequential stream map took: 5.1 s.

Parallel stream map took: 9.5 s.

As you can see, this is a very poor performing implementation. It is extremely inefficient to create a strem of Integer objects and to collect them back again in a list. Performance is abysmal and it uses a buckload of memory. In this case I made the map over 50 M elements (instead of 200 M as I used in the C++ and Rust versions) because over that I exhausted my 16 G memory.

Let’s try using primitive arrays int[] instead of List<Integer> and an IntUnaryOperator instead of UnaryOperator<Integer> to see what the overhead of Integer object type can make.

  public int[] prim_map_seq_stream(int[] arr, IntUnaryOperator foo) {
    return IntStream.of(arr).map(foo).toArray();
  }

  public int[] prim_map_par_stream(int[] arr, IntUnaryOperator foo) throws Exception {
    return pool.submit(() -> IntStream.of(arr).parallel().map(foo).toArray()).get();
  }

Sequential primitive int stream map took: 0.12 s.

Parallel primitive int stream map took: 0.05 s.

Not only is this a factor of ~50 faster, but we also get a decent speedup.

Next attempt attempt is using a simple for loop and divide the work in paralle similar to the threads version in C++. Let’s stick with the primitive arrays as it seems much mode efficient than using lists of Integers.

  public int[] arr_map_seq_for(int[] arr, IntUnaryOperator foo) {
    for (int i = 0; i < arr.length; i++) {
      arr[i] = foo.applyAsInt(arr[i]);
    }
    return arr;
  }

  public int[] arr_map_par_for(int[] arr, IntUnaryOperator foo) {
    int procs = pool.getParallelism();
    List<RecursiveAction> tasks = new ArrayList<>();
    for (int p = 0; p < procs; p++) {
      ArrLocalMap localMap = new ArrLocalMap(arr, procs, p, foo);
      tasks.add(localMap);
      pool.execute(localMap);
    }

    for (RecursiveAction t : tasks) {
      t.join();
    }

    return arr;
  }


  private static class ArrLocalMap extends RecursiveAction {

    int[] arr;
    int procs;
    int p;
    IntUnaryOperator foo;

    public ArrLocalMap(int[] arr, int procs, int p, IntUnaryOperator foo) {
      this.arr = arr;
      this.procs = procs;
      this.p = p;
      this.foo = foo;
    }

    @Override
    protected void compute() {
      int from = p * arr.length / procs;
      int to = p == procs - 1 ? arr.length : (p + 1) * arr.length / procs;
      for (int i = from; i < to; i++) {
        arr[i] = foo.applyAsInt(arr[i]);
      }
    }
  }

My, my! Can you see why people think Java is verbose?

Here, I had to create a special object LocalMap extending the RecursiveAction class which can be executed by the Fork/Join framework. This is the same mechanism that is used by the parallel streams interface so I thought it should be comparable.

Sequential for-loop map took: 0.027 s.

Parallel for-loop map took: 0.024 s.

The sequential version is way faster than the streams version, but I barely get any speedup of the parallel code. Probably because there’s not enough work. For comparison, using List<Integer> instead of int[] is about 100 times slower and uses much more memory and using streams of Integers is yet another 2-3 times slower.

Python

It’s easy to see why Python is so popular with data scientists and others not being software developers as their profession. Here are two sequenial implementations of the map pattern:

def map_for(arr, foo):
    for el in arr:
        el = foo(el)
    return arr

def map_seq(arr, foo):
    return list(map(foo, arr))

The first one simply takes an iterable argument arr and iterates over it and applies the function foo. The second one uses the built in mapfunction.

For loop map: 10.8 s Built in map: 13.7 s.

This is running over an array of 100 M elements. As you know, Python is an interpreted language and therefore it is not meaningful to compare with any of the compiled languages.

Let’s parallelise this using the multiprocessing library. This uses separate processes instead of threads, since Python “suffers” from the global interpreter lock (GIL), making parallel threads infeasible. Luckily, there’s a built in map function also for a pool of processes, but the communication overhead is pretty large.

def map_par(arr, foo, pool):
    return pool.map(foo, arr, 10000)

The code is very simple, as usual with Python, but with an execution time of 23.0 s, there is something not right.

Finally, let’s use the NumPy library:

np_arr = np.array(arr)
np_arr_map = foo(np_arr)

This executed in 0.7 s which approaches the compiled languages. I’m running this on a laptop with an Nvidia GeForce GPU but the NumPy library does not automatically use it. However, it certainly uses optimised versions of the Intel MKL libraries which internaly uses CPU parallelism and vector instructions.

Discussion

Key take-aways from this discussion: Use compiled languages if speed of execution or memory consumption is of importance. Rust and C++ were simply outstanding and there are simple ways to implement the map parallel pattern in these languages. In C++ simply using the standard library is a good thing. I have yet to explore all the various Rust crates supporting parallelism in higher abstraction than using threads, but Rayon made the map pattern particularly simple and basically using the same programming idiom as parallel streams in Java.

Java is problematic when you need really high performance. You need to use the primitive data types and arrays instead of the rich set of collections and this severly limits what you can do in the code. Instead of writing Java code it becomes more like C code in Java syntax.

Finally, Python is a wonderful language to write code in (as long as you do not make mistakes and need to debug as you have no compiler to help you), and with high-performance libraries such as NumPy you basically do not need to sacrifice any performance.
